# Snake and Ladder Game using Python & Tkinter

A personal passion project built using **Python** and **Tkinter**, created during my early programming journey to challenge myself and explore how real GUI-based games work internally.

This project is more than just a simple Snake and Ladder game — it represents my learning process, experimentation, problem-solving skills, and transition from procedural programming to Object-Oriented Programming (OOP).

---

# Project Motivation

When I first started learning Python, I became deeply interested in creating interactive applications and games rather than only writing console-based programs.

To test my understanding of GUI development, I learned the **Tkinter** library and decided to build a complete Snake and Ladder game from scratch.

The goal was not only to make the game work, but also to understand:

- GUI positioning
- Event-driven programming
- Game logic implementation
- Animation-like movement
- Code structuring
- OOP concepts
- Mathematical coordinate calculations

This project became one of my biggest self-learning milestones.

---

# Project Evolution

## Phase 1 — Functional Programming Version

Initially, the entire project was developed using only **functions**.

The game logic, UI handling, dice rolling, player movement, board updates, and event handling were all separated into multiple functions.

This phase helped me understand:

- Function-based architecture
- Tkinter basics
- Code modularization
- State management

---

## Phase 2 — Object-Oriented Programming (OOP) Version

After successfully completing the functional version, I challenged myself further by converting the entire project into an **Object-Oriented Programming structure**.

The project was redesigned using classes and objects to improve:

- Code organization
- Reusability
- Scalability
- Maintainability

This transition helped me practically understand how OOP works in real-world projects.

# Interesting Part of the Project

One of the most challenging aspects of this project was handling the UI positioning.

Tkinter is **not fully responsive**, so every component placement had to be manually calculated.

This includes:

- Board alignment
- Coin movement
- Dice positioning
- Player path calculations
- Coordinate mapping

I manually calculated and mapped pixel positions based on my monitor resolution so that the player tokens move correctly according to the dice value.

A large amount of mathematical positioning and coordinate planning was done manually to make the gameplay visually accurate.

# Important Note About Screen Resolution

Since Tkinter does not automatically handle responsive layouts well, the UI was optimized specifically for my screen resolution.

Because of this:

- The game may not appear perfectly aligned on all systems
- Some UI components may require repositioning
- Users may need to adjust pixel values according to their screen size
